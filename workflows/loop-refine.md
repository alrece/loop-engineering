<purpose>
Loop Engineering 的提示词专业化优化器（v4.3 新增）。在用户用一句话提出需求后，
通过「深度分析 → 动态追问 → 三套生成 → 用户选定」四阶段，把模糊需求转化为
层级清晰、执行顺序明确、防 Agent 越界的专业提示词。

设计哲学：用户给一句话，agent 还三套。不替用户做决定，只把决定结构化呈现。
</purpose>

<principles>
1. **不臆测、必追问**：原始提示词缺失或模糊的关键维度，必须通过 AskUserQuestion 向用户确认，不自行编造
2. **总是生成三套**：无论原始提示词多清晰，都产出「标准 / 精简 / 高阶强约束」三套供用户选
3. **不替用户选**：三套生成后用 AskUserQuestion 让用户选，agent 不得擅自定版
4. **三套内容一致、形式分化**：三套覆盖相同需求，仅结构/详略/约束强度不同
5. **保留追问记录**：用户答复写入产物，可审计、可追溯需求演化
</principles>

<dimensions>
agent 深度分析时，按以下 8 个维度识别原始提示词的缺失/模糊点，并据此生成追问：

| # | 维度 | 判定要点（缺失/模糊即追问） |
|---|------|---------------------------|
| ① | 目标用户 | 未明确「谁用」或过于宽泛（如"所有人"） |
| ② | 核心使用场景 | 未描述「在什么情境下解决什么问题」 |
| ③ | 技术栈偏好 | 未指定语言/框架；或存在多种合理选择 |
| ④ | 规模 / 性能 | 未提及预期用户量、数据量、响应时延 |
| ⑤ | 鉴权 / 安全 | 未说明登录方式、权限分级、数据敏感等级 |
| ⑥ | 优先级 / MVP 范围 | 未区分「必做 vs 可选」；未界定首版交付边界 |
| ⑦ | 既有代码约束 | 未说明是否在既有项目上扩展、有无不可改的红线 |
| ⑧ | 验收标准 | 未定义「做完什么样算合格」的客观判据 |

**追问数量规则**：3-6 个，按缺失维度数决定；提示词越模糊问题越多，最多 6 个；
若某维度用户已在原始提示词中明确回答，则不再追问。
</dimensions>

<process>

<step name="receive">
接收原始提示词（来自 `$ARGUMENTS` 或上一环节传入）。若提示词为空，提示用户：
`请提供你的需求描述。例：/loop:refine "做一个团队协作的待办应用"`，并停止。
</step>

<step name="analyze">
按 `<dimensions>` 的 8 个维度深度分析原始提示词：
- 已明确：标记 ✓ 并摘录原文证据
- 部分明确：标记 △ 指出模糊点
- 缺失：标记 ✗

输出内部判定表（不直接展示给用户，供下一步生成问题用）。
</step>

<step name="generate_questions">
基于分析结果，**动态生成** 3-6 个追问问题，规则：
- 每个缺失（✗）或模糊（△）维度生成 1 个问题
- 问题表述贴合该维度，避免泛泛（例：「目标用户是谁？」→「核心使用者是个人用户、小团队还是企业？他们的主要职责是什么？」）
- 若缺失维度 < 3，补充最关键的隐性假设问题凑足 3 个（如「这次是要新建项目还是在既有代码上扩展？」）
- 不超过 6 个
</step>

<step name="ask_user">
用 **AskUserQuestion 工具一次性追问所有问题**（多问题并行，单次交互）：
- 每个问题提供 2-4 个候选选项 + 允许用户选"其他"自定义
- 选项设计原则：覆盖常见场景、互斥、可操作
- header 用维度简称（≤12 字符，如"目标用户""技术栈"）
</step>

<step name="synthesize">
深度思考，综合「原始提示词 + 追问答复」，**同时生成三套提示词**。
三套必须覆盖相同的需求事实，仅结构/详略/约束强度不同（见 `<variants>`）。
</step>

<step name="present">
展示三套提示词给用户，含每套的**适用场景**和**关键差异**：
- 标准版：日常使用，层级逻辑清晰、执行顺序明确
- 精简版：迭代对话快速下发，适合多轮交互
- 高阶强约束版：适配 AI Agent，强化文件读取、防越界

用 **AskUserQuestion 让用户选一套**（选项：标准 / 精简 / 高阶强约束）。
agent 不得替用户选，不得默认标准版。
</step>

<step name="persist">
用户选定后：
1. 写入 `.loop/refined-prompt.md`（格式见 `<output>`）—— 含原始提示词、追问记录、三套全文、用户选定的版本
2. 更新 `.loop/STATE.yaml`：若 `artifacts.refine` 字段存在则指向产物路径
3. 追加 `.loop/timeline.jsonl`：`{"ts","phase","step":"refine","event":"complete"}`
</step>

<step name="handoff">
提示用户下一步：
```
✓ 提示词已优化并保存到 .loop/refined-prompt.md（{选定版本}版）

下一步建议：
  → /office-hours（带优化后提示词进入构想深挖）
  → 或 /loop:run --next --auto（自动推进闭环）
```
不自动调用，由用户决定。
</step>

</process>

<variants>
三套提示词模板。生成时按下方结构填充，**三套的需求事实必须一致**。

### 变体 1：标准版（日常使用）
────────────────────────────────────────
结构层级清晰，执行顺序明确，适合作为项目主需求文档。

```
# {项目/需求名}

## 项目目标
{一句话目标 + 解决什么问题}

## 目标用户与场景
- 目标用户：{角色 + 职责}
- 核心场景：{在什么情境下做什么}
- 次要场景：{如有}

## 核心需求
### 功能需求
1. {需求}（优先级：P0/P1/P2）
2. ...

### 非功能需求
- 性能：{用户量/数据量/响应时延}
- 安全：{鉴权方式 + 权限分级}
- 兼容性：{如有}

## 技术栈
- 语言/框架：{明确指定}
- 数据存储：{如有}
- 部署：{如有}

## 执行顺序
1. {先做什么}
2. {再做什么}
3. ...

## 验收标准
- [ ] {可客观判定的标准 1}
- [ ] {可客观判定的标准 2}
- [ ] ...

## 范围边界（不做）
- {明确排除项}
```

### 变体 2：精简版（迭代对话，快速下发）
────────────────────────────────────────

### 变体 3：高阶强约束版（AI Agent 强化防护）
────────────────────────────────────────
适配 AI Agent 执行，强化文件读取、鉴权保护、原有逻辑保护、质量门强制。
条款详细写法见 `<strict_clauses>`。

```
# {项目/需求名}（高阶强约束版）

## 🔒 强制读取文件清单（agent 必读，跳读即停）
- {必读文件 1}（读取目的）
- {必读文件 2}（读取目的）
- README.md（项目约定）
- AGENTS.md（行为规范）
- {OpenSpec specs / 核心模块路径}

## 🚫 红线条款（违反即停）
### 鉴权逻辑保护
{见 strict_clauses.auth}

### 原有逻辑保护
{见 strict_clauses.legacy}

## 项目目标
{一句话目标}

## 核心需求（P0 必做）
1. {需求}
2. {需求}

## 执行约束
### 质量门强制执行
{见 strict_clauses.quality}

### 变更范围限定
- 允许修改：{模块/文件白名单}
- 禁止修改：{模块/文件黑名单}

## 验收标准
- [ ] {客观标准}

## 范围边界（不做）
- {排除项}
```

<strict_clauses>
高阶强约束版的四类条款详细写法。生成时根据具体项目填充 `{占位}`。

### 1. 鉴权逻辑保护（auth）
```
- 不得跳过、绕过或禁用任何鉴权中间件、权限校验、会话验证逻辑
- 不得删除或降级现有 auth/permission/guard 相关代码，除非本提示词明确授权
- 新增接口必须继承现有鉴权基类/装饰器，不得裸写无鉴权端点
- 涉及用户身份的变更（注册/登录/改密/删除账号）必须有完整审计日志
- 敏感操作（删除/导出/权限变更）必须二次确认 + 权限复核
```

### 2. 原有逻辑保护（legacy）
```
- 修改任何既有函数/模块前，必须先读完其完整实现 + 相关注释/文档，理解原设计意图
- 不得擅自重构、改名、调整既有公共接口的签名，除非本提示词明确要求
- 既有测试用例必须保持通过；若需改动测试，必须在 PR 说明中列出理由
- 行为变更必须显式标注（diff 中标出"行为变更"），不得静默改变既有行为
- 数据库 schema 变更必须通过 migration，不得直接改库
```

### 3. 质量门强制执行（quality）
```
- 每个 plan 完成后必须跑 adversarial_gate，不得跳过
- verify-security 必须 0 个 critical+high 才放行，不得用 --force 绕过
- 产物必须符合 spec，spec 变更必须走 OpenSpec 变更流程
- 提交前必须跑完整测试套件，测试不过不得提交
- 对抗门 escalate 时必须停下问人，不得自行降级
```

### 4. 强制文件读取清单（readlist）
agent 按项目实际情况生成，至少包含：
- 项目说明类：README.md、AGENTS.md、CONTRIBUTING.md（若有）
- 架构类：DESIGN.md、docs/architecture.md（若有）
- 规格类：openspec/changes/、openspec/specs/（若用 OpenSpec）
- 核心模块：与本次需求直接相关的源文件路径（agent 扫描后列出）
- 配置类：package.json/pyproject.toml/go.mod 等（确认技术栈与依赖）
每项后注明读取目的，让 agent 知道为什么要读。
</strict_clauses>

<state_integration>
与 Loop Engineering 状态机的集成：
- refine 是**独立环节**，不强制要求 `.loop/` 已 init（无 STATE.yaml 时仅在 cwd 生成 refined-prompt.md）
- 若 `.loop/STATE.yaml` 存在：更新 `artifacts.refine` 指向产物路径，追加 timeline
- refine 产出可被 `/office-hours`（Phase 1）读取作为深挖起点
- refine 不改变 current_phase/step（它是横切工具，不推进闭环）
</state_integration>

<auto_mode>
**--auto 模式行为**：refine 是交互式工具，`--auto` 下仍保留两次 AskUserQuestion（追问 + 选版），
因为这两次交互是 refine 的核心价值。但 agent 不在其他环节停——追问和选版后直接进入 handoff，
不再因"是否调 office-hours"停问（仅提示，由用户自主决定）。

**--interactive 模式行为**：每步都停，确认后再继续。
</auto_mode>
